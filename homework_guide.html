DevOps Homework Guide

Understand what each homework does, why it works that way, and every mistake you might hit — real and hypothetical.

v3.x

CloudFormation + UserData

One command deploys everything — infra AND app — in a single CloudFormation template

🏠 Metaphor: Think of it like ordering a pre-furnished apartment. You place one order (CloudFormation) and the moving crew (UserData script) automatically sets up the furniture when the apartment is ready. You don't supervise — it all runs automatically on first boot.

💡 What is CloudFormation?

CloudFormation is AWS's way of creating infrastructure using a config file (YAML) instead of clicking through the AWS website. You describe what you want — a server, a network, firewall rules — and AWS creates it all automatically.

Think of it as a recipe: you write the recipe once, and AWS cooks it every time.

📜 What is UserData?

UserData is a bash script that runs automatically the very first time an EC2 server boots up. It's embedded inside the CloudFormation template.

In v3.x, this script does everything: installs PostgreSQL, installs Node.js, clones LobeChat, builds it, and starts it. One script, one boot, full deployment.

⚠️ The script only runs once — on first boot. If something breaks, you can't re-run it automatically.

🔄 Workflow — Step by Step

1
Get AWS credentials
Copy from sandbox portal → write to ~/.aws/credentials
2
Create SSH key pair
So you can SSH into the server later if needed
aws ec2 create-key-pair --key-name lobechat-key --query 'KeyMaterial' --output text > ~/.ssh/lobechat-key.pem chmod 400 ~/.ssh/lobechat-key.pem
3
Deploy CloudFormation stack
This creates the VPC, subnet, security group, EC2 instance — AND embeds the UserData script. Takes ~3-4 min.
aws cloudformation deploy \ --template-file infra/cloudformation.yml \ --stack-name lobechat \ --capabilities CAPABILITY_IAM
4
UserData runs automatically on first boot
The EC2 instance starts, runs the bash script — installs PostgreSQL, pg_search, Node.js, pnpm, bun, clones LobeChat, builds it (SPA + Next.js), migrates DB, creates systemd service. Takes ~20 min. You can watch it with:
ssh -i ~/.ssh/lobechat-key.pem ubuntu@<PUBLIC_IP> sudo tail -f /var/log/user-data.log
5
Access LobeChat
Get the public IP from CF outputs, open in browser
http://<PUBLIC_IP>:3210
6
Take screenshots & delete stack
Browser screenshot + then delete everything
aws cloudformation delete-stack --stack-name lobechat aws cloudformation wait stack-delete-complete --stack-name lobechat
Why v3.x exists

v1.x and v2.x required manual steps or clicking in AWS. v3.x is the first fully automated deployment — one command and walk away. The downside is that everything is crammed into one giant bash script, which is hard to debug and only runs once.

🐛 Bugs & Mistakes

Real Bug PostgreSQL auto-starts — systemctl start is a no-op ▶
Symptom

CREATE EXTENSION pg_search fails with "pg_search not found". DB migration fails. App never starts.

Root Cause

When you run apt-get install postgresql-16, Postgres auto-starts immediately. The script then adds shared_preload_libraries = 'pg_search' to the config file. Then it runs systemctl start postgresql — but Postgres is already running, so this is silently ignored. pg_search never loads.

Fix

Use systemctl restart postgresql instead of systemctl start. Restart forces Postgres to reload its config and actually load pg_search.

Real Bug set -e silently kills the script — you think it worked but it didn't ▶
Symptom

CloudFormation says stack is created successfully. You open the browser — nothing works. LobeChat is not installed at all.

Root Cause

The UserData script starts with set -e — this means any single command that fails causes the whole script to stop immediately. CloudFormation doesn't know the script failed; it just sees the EC2 instance running. If DB migration fails at step 15, steps 16-30 (creating the systemd service, starting the app) are silently skipped.

Fix

SSH in and check the logs:

sudo cat /var/log/user-data.log

Find where the script stopped, fix the underlying issue (e.g. restart Postgres, re-run migration manually), then manually complete the remaining steps.

Real Bug LobeChat shows blank white page after login (tanstack build missing) ▶
Symptom

Login works, but after logging in you see a completely white/blank page. No errors visible on screen.

Root Cause

@tanstack/react-query@5.94.4 was published to npm with a broken package — the build/ directory is missing. Vite crashes during SPA build. Also, if you only ran pnpm run build (Next.js) without first running build:spa, the SPA is missing entirely.

Fix

1. Download working tanstack builds from npm, copy into the broken installation. 2. Always run builds in correct order: build:spa → build:spa:copy → build.

Hypothetical EC2 runs out of disk space during build ▶
Symptom

UserData log shows "No space left on device" during pnpm install or build. Script stops. App not deployed.

Root Cause

If you set VolumeSize too small (under 20GB), the pnpm node_modules + Next.js build artifacts + pg_search package can fill the disk. node_modules alone can be 3-4GB.

Fix

Delete the stack and redeploy with a larger volume size. In the CF template, increase VolumeSize default to 30GB. Or SSH in and run df -h to confirm, then docker system prune / apt-get clean to free space before retrying.

Hypothetical GitHub is down / LobeChat repo clone fails ▶
Symptom

UserData log shows "fatal: repository not found" or timeout during git clone. Everything after clone is skipped (set -e). App not deployed.

Root Cause

The script does git clone https://github.com/lobehub/lobe-chat. If GitHub is unreachable (network issue, DNS problem, or GitHub outage), the clone fails and set -e stops the whole script.

Fix

SSH in, check /var/log/user-data.log, manually run the git clone and subsequent commands from where the script stopped. Alternatively, add a retry loop in the UserData script: for i in 1 2 3; do git clone ... && break || sleep 10; done

v4.x

CloudFormation + Docker Compose

Same CF infra — but instead of building the app from source, run pre-built Docker containers

📦 Metaphor: v3.x was cooking from scratch — buy ingredients, chop, cook. v4.x is ordering meal-kit boxes (Docker images): everything is pre-packaged, you just heat and serve. Much faster, much less can go wrong during "cooking."

🐳 What is Docker?

Docker packages an app and everything it needs (code, runtime, libraries) into a container — a self-contained box that runs identically anywhere. No "it works on my machine" problems.

A Docker image is the blueprint. A container is the running instance. Images are stored on registries like Docker Hub (docker.io) or GitHub Container Registry (ghcr.io).

🎼 What is Docker Compose?

When you need multiple containers that work together (LobeChat + PostgreSQL + MinIO), Docker Compose lets you define them all in one docker-compose.yml file and start them all with docker compose up -d.

It also handles startup order — it waits for Postgres to be healthy before starting LobeChat, so the app doesn't crash trying to connect to a database that isn't ready yet.

🔄 Workflow — Step by Step

1
Get AWS credentials + create key pair
Same as v3.x
2
Deploy CloudFormation stack
CF creates VPC, subnet, SG, EC2. UserData installs Docker + writes docker-compose.yml + runs docker compose up -d
aws cloudformation deploy \ --template-file infra/cloudformation.yml \ --stack-name lobechat \ --capabilities CAPABILITY_IAM
3
Docker pulls images and starts containers
~2 minutes to pull lobehub/lobe-chat, pgvector/pgvector:pg16, minio/minio — then all 4 containers start. No compilation needed.
4
Access LobeChat + take screenshots + delete
Same as v3.x
http://<PUBLIC_IP>:3210
Why v4.x is better than v3.x

v3.x took ~20 min to build LobeChat from source (and could fail at any step). v4.x takes ~2 min — just downloading pre-built images. No tanstack bug. No SPA build order. No pg_search compilation. The official Docker image lobehub/lobe-chat handles everything internally.

🐛 Bugs & Mistakes

Real Bug Teacher's Docker image is private — EC2 gets 403 ▶
Symptom

UserData log shows: Error response from daemon: unauthorized: authentication required. LobeChat container never starts.

Root Cause

The template used ghcr.io/oriolrius/lobechat — the teacher's private image on GitHub Container Registry. EC2 instances have no GitHub credentials, so every pull attempt returns 403 Forbidden.

Fix

Replace with the official public image on Docker Hub. Change in docker-compose.yml:

# Before: image: ghcr.io/oriolrius/lobechat:latest # After: image: lobehub/lobe-chat:latest

Docker Hub images are public by default — no auth needed.

Real Bug chmod 400 prevents overwriting the .pem key file ▶
Symptom

bash: /root/.ssh/lobechat-key.pem: Permission denied when trying to create the key pair again.

Root Cause

The first time you create the key, you run chmod 400 on the .pem file (read-only, no write). The next time you try to create a new key and redirect to the same file, Linux refuses because the file is not writable — even by the owner.

Fix

Delete the old key file before creating a new one:

rm -f ~/.ssh/lobechat-key.pem aws ec2 create-key-pair --key-name lobechat-key \ --query 'KeyMaterial' --output text > ~/.ssh/lobechat-key.pem chmod 400 ~/.ssh/lobechat-key.pem
Hypothetical LobeChat can't connect to Postgres — uses "localhost" instead of container name ▶
Symptom

LobeChat container keeps crashing with "connection refused" errors in docker compose logs lobechat. Postgres container is running fine.

Root Cause

In Docker Compose, containers talk to each other using their service names as hostnames — not localhost. If DATABASE_URL is set to postgresql://postgres:pass@localhost:5432/lobechat, LobeChat looks for Postgres on its own container (nothing there). The correct hostname is the service name: postgres.

Fix

Change DATABASE_URL to use the service name:

DATABASE_URL=postgresql://postgres:password@postgres:5432/lobechat # ^^^^^^^^ # This is the service name, not localhost
Hypothetical File uploads work internally but fail from browser (S3_ENDPOINT vs S3_PUBLIC_DOMAIN) ▶
Symptom

You can upload files in LobeChat but can't view them. Images don't load. The URL in the browser tries to reach http://minio:9000/... which doesn't work outside Docker.

Root Cause

Two separate env vars control MinIO access: S3_ENDPOINT is used by LobeChat server-side (inside Docker, uses service name minio). S3_PUBLIC_DOMAIN is used by the browser to load files (must use the real public IP). If both are set to the service name, the browser can't reach MinIO.

Fix

Set them separately:

S3_ENDPOINT=http://minio:9000 # server-to-server (inside Docker) S3_PUBLIC_DOMAIN=http://PUBLIC_IP:9000 # browser-to-minio (public)
v5.x

CloudFormation + Ansible

CloudFormation creates infra only — Ansible configures and installs the app from your laptop

🏗️ Metaphor: CloudFormation is the construction crew that builds the empty building (server, network, security). Ansible is the interior designer who comes in afterwards and sets up every room (installs software, writes config files, starts services) — from a checklist on their laptop. Two separate jobs, two separate teams.

🤖 What is Ansible?

Ansible is a tool that runs a playbook (a list of tasks written in YAML) on remote servers over SSH. You run it from your laptop — it connects to the server and executes each task one by one.

Key properties:

• Idempotent — running it twice gives the same result. If something is already installed, it skips it. Safe to re-run if something fails.

• Readable — tasks are human-readable YAML: "install postgresql", "copy this file", "start this service"

• No agent needed — only needs SSH access to the server

📋 Key Files

ansible/playbook.yml — the recipe: installs PostgreSQL, MinIO, Node.js, builds LobeChat, creates systemd services

ansible/inventory.yml — tells Ansible which server to connect to (IP address, SSH key, username). Generated from inventory.yml.template — never committed to git.

🔄 Workflow — Step by Step

1
Get AWS credentials + write to ~/.aws/credentials
⚠️ Must write to file — not just export in terminal
2
Create SSH key pair
Check it doesn't already exist first
aws ec2 describe-key-pairs --key-names lobechat-key 2>/dev/null || \ aws ec2 create-key-pair --key-name lobechat-key \ --query 'KeyMaterial' --output text > ~/.ssh/lobechat-key.pem chmod 400 ~/.ssh/lobechat-key.pem
3
Deploy CloudFormation stack (infra only)
Creates server + network. No UserData app install. Takes ~3-4 min.
aws cloudformation deploy \ --template-file infra/cloudformation.yml \ --stack-name lobechat \ --capabilities CAPABILITY_IAM
4
Get public IP + create inventory file
Generate the Ansible inventory from the template, replacing <PUBLIC_IP>
PUBLIC_IP=$(aws cloudformation describe-stacks --stack-name lobechat \ --query 'Stacks[0].Outputs[?OutputKey==`PublicIP`].OutputValue' --output text) sed "s/<PUBLIC_IP>/$PUBLIC_IP/" ansible/inventory.yml.template > ansible/inventory.yml
5
Wait 1-2 min for EC2 to boot
SSH service needs to be ready before Ansible can connect
6
Run Ansible playbook (~15 min)
Ansible installs everything over SSH. You watch the tasks scrolling by in real time.
uv run ansible-playbook -i ansible/inventory.yml ansible/playbook.yml
7
Check PLAY RECAP → open browser → screenshot → delete
PLAY RECAP must show failed=0. Then open LobeChat, take screenshots, delete stack.
Why v5.x is the most professional approach

Clear separation of concerns: CloudFormation owns infrastructure (servers, networks), Ansible owns configuration (what runs on those servers). You can change the app without touching infrastructure, or scale infrastructure without rewriting app config. Ansible's idempotency also means you can safely re-run if something fails — no need to rebuild from scratch.

🐛 Bugs & Mistakes

Real Bug AWS credentials not found — Ansible/AWS CLI commands fail silently ▶
Symptom

aws cloudformation deploy returns "Unable to locate credentials". Or works in one terminal window but not another.

Root Cause

AWS credentials set with export AWS_ACCESS_KEY_ID=... only exist in that specific terminal session. Every new shell (including every bash command Claude Code runs) is a fresh shell with no environment variables.

Fix

Write credentials to ~/.aws/credentials so the AWS CLI picks them up automatically in any shell:

[default] aws_access_key_id = YOUR_KEY aws_secret_access_key = YOUR_SECRET aws_session_token = YOUR_TOKEN region = eu-west-1
Real Bug lobechat-key already exists in AWS — key pair creation fails ▶
Symptom

aws ec2 create-key-pair returns "InvalidKeyPair.Duplicate: The keypair already exists". Your local .pem file may be from a different key (previous session) and SSH will fail.

Root Cause

The key was created in a previous session and never deleted. AWS still has it registered. If you try to create it again, AWS rejects it — and you're now using a .pem file that doesn't match the registered key.

Fix

Delete the old key from AWS and locally, then recreate:

aws ec2 delete-key-pair --key-name lobechat-key rm -f ~/.ssh/lobechat-key.pem aws ec2 create-key-pair --key-name lobechat-key \ --query 'KeyMaterial' --output text > ~/.ssh/lobechat-key.pem chmod 400 ~/.ssh/lobechat-key.pem
Real Bug Ansible fails immediately — SSH connection refused ▶
Symptom

Ansible starts, immediately fails with UNREACHABLE! ... Connection refused or timed out on every task.

Root Cause

EC2 was just created and is still booting. The OS isn't fully loaded yet, the SSH daemon hasn't started. If you run Ansible immediately after aws cloudformation deploy completes, you'll often hit this.

Fix

Wait 1-2 minutes after the CF stack is created before running Ansible. You can verify SSH is ready first:

ssh -i ~/.ssh/lobechat-key.pem -o ConnectTimeout=5 ubuntu@<IP> "echo ready"

If it echoes "ready", Ansible is safe to run.

Hypothetical inventory.yml not created — Ansible can't find any hosts ▶
Symptom

uv run ansible-playbook -i ansible/inventory.yml ansible/playbook.yml fails with "Could not match supplied host pattern" or "No hosts matched".

Root Cause

ansible/inventory.yml is gitignored — it's not in the repo. It must be created from inventory.yml.template by replacing <PUBLIC_IP>. If you forgot this step, Ansible has no idea which server to connect to.

Fix

Generate the inventory file from the template:

PUBLIC_IP=$(aws cloudformation describe-stacks --stack-name lobechat \ --query 'Stacks[0].Outputs[?OutputKey==`PublicIP`].OutputValue' \ --output text) sed "s/<PUBLIC_IP>/$PUBLIC_IP/" ansible/inventory.yml.template > ansible/inventory.yml cat ansible/inventory.yml # verify it looks right
Hypothetical Running ansible-playbook directly instead of uv run ▶
Symptom

ansible-playbook: command not found. Or if Ansible is installed globally, it runs with the wrong Python version and some tasks fail unexpectedly.

Root Cause

In v5.x, Ansible is managed by uv (a fast Python package manager). The correct command is uv run ansible-playbook .... Running ansible-playbook directly looks for a global install that may not exist, or may be a different version.

Fix

Always prefix with uv run:

# Wrong: ansible-playbook -i ansible/inventory.yml ansible/playbook.yml # Correct: uv run ansible-playbook -i ansible/inventory.yml ansible/playbook.yml
DIFF

How They Compare

Same goal (deploy LobeChat), very different approaches

🧭 The Core Idea of Each

All three deploy LobeChat to AWS. The difference is how much automation and where the logic lives:

v3.x Everything in one big bash script inside CloudFormation. Runs on first boot. Hard to debug, runs once, takes 20+ min.

v4.x Still all in CloudFormation UserData, but swaps "build from source" for "pull pre-built Docker images". Faster (2 min), more reliable, but still one script.

v5.x Split into two tools: CloudFormation (infra) + Ansible (app config). Ansible runs from your laptop. Slower to set up but idempotent, readable, and re-runnable.

Feature v3.x v4.x v5.x
Infra tool CloudFormation CloudFormation CloudFormation
App install tool UserData (bash) UserData (Docker) Ansible (from laptop)
App runs as systemd service Docker containers systemd service
Build from source Yes (~20 min) No (pre-built) Yes (~15 min)
Total deploy time ~25 min ~5 min ~20 min
Can re-run if fails No (runs once) No (runs once) Yes (idempotent)
Debug visibility /var/log/user-data.log /var/log/user-data.log Live in terminal
Where you run it from Your laptop (1 cmd) Your laptop (1 cmd) Your laptop (2 cmds)
Separation of concerns None None Clean (infra vs app)
Industry usage Simple projects Most common Enterprise standard

🎯 The Key Difference: Where Does the Script Run?

v3.x & v4.x UserData runs on the server during first boot. You have no visibility while it runs. If it fails, you SSH in and look at logs after the fact.

v5.x Ansible runs on your laptop over SSH. You watch every task in real time in your terminal. If task 23 of 46 fails, you know exactly which task and why — immediately.

🔁 The Key Difference: What Happens If Something Fails?

v3.x & v4.x UserData runs once. If step 15 fails, steps 16-30 are skipped forever. You either SSH in and manually fix it, or delete the whole stack and start over.

v5.x Ansible is idempotent. Fix the error, re-run the same command. Ansible skips everything already done and only runs the failed task again.

📦 v3.x vs v4.x — Why Docker is Faster

v3.x installs Node.js, pnpm, clones 100k+ files from GitHub, runs pnpm install (downloads thousands of npm packages), compiles two Vite builds + one Next.js build. Each of these can fail.

v4.x downloads one Docker image that already has everything compiled inside. Like downloading a ready-made executable vs compiling from source code.
